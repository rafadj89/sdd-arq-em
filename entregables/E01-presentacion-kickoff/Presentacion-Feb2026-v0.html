Arquitectura de Datos & Data Lake (On-Premise)

Gobierno integral de datos para la plataforma de electromovilidad. El Data Lake centraliza toda la información de integraciones (CPOs, Cárgame, OCPI) con mecanismos asíncronos que garantizan desacople, trazabilidad y calidad de datos.

📥 Zona Raw (Landing)

Ingesta cruda sin transformación. Preserva el dato original para auditoría y reprocesamiento.

  • ▸Reportes OCPI 2.2.1 (JSON original)
  • ▸Respuestas de validación Cárgame
  • ▸Logs de telemetría de estaciones
  • ▸Formato: Parquet / JSON particionado por fecha
Storage: MinIO (S3-compatible) con retención de 5 años

🔄 Zona Curada (Processed)

Datos validados, normalizados y enriquecidos. Listos para consumo analítico y operacional.

  • ▸Esquema unificado de estaciones y conectores
  • ▸Registro maestro de CPOs (enriquecido con Cárgame)
  • ▸Series de tiempo normalizadas (TimescaleDB)
  • ▸Deduplicación y control de calidad automático
ETL: Apache Spark / dbt para transformaciones batch

📊 Zona Analítica (Serving)

Capa de consumo optimizada para dashboards, reportes regulatorios y APIs de datos abiertos.

  • ▸Vistas materializadas para KPIs en tiempo real
  • ▸Reportes UPME / Resolución 40559
  • ▸API Open Data para consulta pública
  • ▸Catálogo de datos con Apache Atlas
Serving: PostgreSQL + Redis caché + API Gateway

⚡ Mecanismos Asíncronos

Patrones de integración asíncrona que garantizan desacople entre sistemas, tolerancia a fallos y procesamiento escalable.

📨

Apache Kafka (Event Streaming)

Bus de eventos central. Tópicos dedicados: raw-reports, cpo-validations, cargame-responses, data-quality-alerts. Retención de 30 días para reprocesamiento.

🔁

Dead Letter Queue (DLQ)

Mensajes que fallan validación o procesamiento se envían a la DLQ para revisión manual o reprocesamiento automático con backoff exponencial.

🕐

CQRS + Event Sourcing

Separación de comandos (escritura) y consultas (lectura). Cada cambio de estado de un CPO o estación se registra como evento inmutable para trazabilidad completa.

🔗

Webhooks & Callbacks Asíncronos

Notificaciones push a CPOs cuando su validación en Cárgame se completa. Patrón request-reply asíncrono con correlation ID para trazabilidad end-to-end.

🏛️ Gobierno de Datos

Marco de gobernanza que asegura calidad, linaje, seguridad y cumplimiento normativo sobre todos los datos de la plataforma.

CATÁLOGO Apache Atlas / DataHub

Catálogo centralizado de metadatos. Clasificación automática de datos sensibles (PII) de CPOs y usuarios. Búsqueda semántica de datasets.

LINAJE Data Lineage End-to-End

Trazabilidad completa desde la fuente (CPO / Cárgame) hasta el reporte final. Cada transformación queda registrada con timestamp, versión y responsable.

CALIDAD Great Expectations / dbt Tests

Validaciones automáticas: completitud, unicidad, rangos válidos, consistencia referencial. Alertas en Slack/Teams cuando un umbral de calidad se degrada.

SEGURIDAD RBAC + Cifrado en Reposo

Control de acceso basado en roles (RBAC) por zona del Data Lake. Cifrado AES-256 en reposo. Enmascaramiento dinámico de datos sensibles para entornos no-productivos.

Plan de Recuperación ante Desastres (DRP)

Estrategia de respaldo cross-region en OCI con failover automatizado. Garantiza continuidad del servicio para la plataforma de electromovilidad.

PRIMARIA

Bogotá Region (GRU / Santiago)

▸OKE Cluster: 3 nodos HA, multi-AD
▸Autonomous DB: Active, auto-backup cada 1h
▸Object Storage: Replicación cross-region activa
▸OCI Streaming: Kafka managed, 3 brokers
▸OCI Vault: Keys replicadas a región DR
DR STANDBY

Región Secundaria (Vinhedo / Ashburn)

▸OKE Cluster: Standby (warm), escala en failover
▸Autonomous DB: Data Guard Standby, sync automático
▸Object Storage: Réplica read-only, failover automático
▸OCI Full Stack DR: Orquestación de failover automatizada
▸DNS Failover: OCI Health Checks + Traffic Steering

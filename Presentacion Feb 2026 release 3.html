⚡ Pipeline CI (Continuous Integration)

OCI DevOps Build Pipelines — Trigger automático en cada PR

1
Lint & Format: ESLint, Prettier, golangci-lint, Black/Ruff
2
Unit Tests: Vitest (Frontend), pytest (Python), go test (Go) — cobertura > 80%
3
SAST: SonarQube — análisis estático de código, vulnerabilidades
4
Build & Push: Docker multi-stage build → OCIR (vulnerability scan)
5
Integration Tests: Tests contra servicios mock (Cárgame, OCPI)
Approval Gate: 1 Sr + Arquitecto para develop. 2 approvals para main.

🚀 Pipeline CD (Continuous Deployment)

ArgoCD GitOps — Deployment declarativo multi-ambiente

1
DEV: Auto-sync en merge a develop — deploy automático sin approval
2
QA: Manual sync — requiere approval de QA Lead. Tests E2E + OCPI compliance automáticos
3
SANDBOX: Manual sync — approval Arquitecto + DevOps Sr. DAST (OWASP ZAP) previo
4
PROD: Manual sync — approval Arquitecto + DevOps Sr + Seguridad IT. Canary deployment 10% → 50% → 100%
Rollback: ArgoCD mantiene historial de deployments. Rollback en <2 min a versión anterior.

📖 Manual de Integración CPO

Flujo detallado de cómo un CPO se integra a la plataforma de interoperabilidad UPME. Ver especificaciones completas en /specs/guia-integracion-cpo.md

1

Registro en Cárgame

▸CPO se registra como operador en el sistema Cárgame
▸Validación documental y regulatoria por Cárgame / MinEnergía
▸Estado resultado: ACTIVE o REJECTED
// Validación automática UPME → Cárgame
GET /api/v1/cpo/{nit}/status
→ { "registered": true, "status": "ACTIVE" }
Baja de CPO:

Si Cárgame cambia status a INACTIVE/SUSPENDED → Se revocan credenciales automáticamente. El CPO recibe notificación con proceso de apelación. Debe re-certificarse para reactivación.

2

Conexión & Capa de Seguridad

▸Generación de API Key única por CPO
▸Emisión de certificado mTLS (X.509) para autenticación mutua
▸Credenciales OAuth2 (client_id / client_secret)
▸Registro de IPs públicas del CPO (máx. 10 CIDR)
Capas de seguridad (en orden):
WAF → IP Whitelist → mTLS → JWT → API GW → Servicio
Filtrado por IP Pública:

Decisión: Opcional en Fase 1 (mTLS + JWT suficiente), obligatorio en Fase 2 con período de gracia de 90 días. VPN Connect disponible para CPOs sin IP fija.

3

Rate Limits & SLA

Recurso Límite
PUT /locations (ingesta)120 req/min
GET /locations (consulta)300 req/min
POST /token (auth)10 req/min
Global por CPO1,000 req/min
Tiers por tamaño de CPO:
Pequeño (1-20 est.): 500/min
Mediano (21-100): 1,000/min
Grande (101-500): 2,500/min
Enterprise (500+): 5,000/min

Arquitectura de Datos & Data Lake (On-Premise)

Gobierno integral de datos para la plataforma de electromovilidad. El Data Lake centraliza toda la información de integraciones (CPOs, Cárgame, OCPI) con mecanismos asíncronos que garantizan desacople, trazabilidad y calidad de datos.

📥 Zona Raw (Landing)

Ingesta cruda sin transformación. Preserva el dato original para auditoría y reprocesamiento.

  • ▸Reportes OCPI 2.2.1 (JSON original)
  • ▸Respuestas de validación Cárgame
  • ▸Logs de telemetría de estaciones
  • ▸Formato: Parquet / JSON particionado por fecha
Storage: MinIO (S3-compatible) con retención de 5 años

🔄 Zona Curada (Processed)

Datos validados, normalizados y enriquecidos. Listos para consumo analítico y operacional.

  • ▸Esquema unificado de estaciones y conectores
  • ▸Registro maestro de CPOs (enriquecido con Cárgame)
  • ▸Series de tiempo normalizadas (TimescaleDB)
  • ▸Deduplicación y control de calidad automático
ETL: Apache Spark / dbt para transformaciones batch

📊 Zona Analítica (Serving)

Capa de consumo optimizada para dashboards, reportes regulatorios y APIs de datos abiertos.

  • ▸Vistas materializadas para KPIs en tiempo real
  • ▸Reportes UPME / Resolución 40559
  • ▸API Open Data para consulta pública
  • ▸Catálogo de datos con Apache Atlas
Serving: PostgreSQL + Redis caché + API Gateway

⚡ Mecanismos Asíncronos

Patrones de integración asíncrona que garantizan desacople entre sistemas, tolerancia a fallos y procesamiento escalable.

📨

Apache Kafka (Event Streaming)

Bus de eventos central. Tópicos dedicados: raw-reports, cpo-validations, cargame-responses, data-quality-alerts. Retención de 30 días para reprocesamiento.

🔁

Dead Letter Queue (DLQ)

Mensajes que fallan validación o procesamiento se envían a la DLQ para revisión manual o reprocesamiento automático con backoff exponencial.

🕐

CQRS + Event Sourcing

Separación de comandos (escritura) y consultas (lectura). Cada cambio de estado de un CPO o estación se registra como evento inmutable para trazabilidad completa.

🔗

Webhooks & Callbacks Asíncronos

Notificaciones push a CPOs cuando su validación en Cárgame se completa. Patrón request-reply asíncrono con correlation ID para trazabilidad end-to-end.

🏛️ Gobierno de Datos

Marco de gobernanza que asegura calidad, linaje, seguridad y cumplimiento normativo sobre todos los datos de la plataforma.

CATÁLOGO Apache Atlas / DataHub

Catálogo centralizado de metadatos. Clasificación automática de datos sensibles (PII) de CPOs y usuarios. Búsqueda semántica de datasets.

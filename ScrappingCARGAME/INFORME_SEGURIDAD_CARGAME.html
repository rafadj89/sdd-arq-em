🛡️ Evaluación OWASP API Security Top 10 (2023)

Análisis contra el estándar internacional de seguridad en APIs de OWASP (Open Worldwide Application Security Project).

VIOLADO

API1:2023 — Broken Object Level Authorization (BOLA)

Evidencia: Todos los endpoints devuelven TODOS los registros sin filtro por usuario autenticado. El token JWT tiene rol "intelligence" con acceso total a toda la base de datos.

Remediación: Implementar control de acceso por objeto. Cada usuario solo debe ver sus propios datos o datos autorizados.

VIOLADO

API2:2023 — Broken Authentication

Evidencia: Token JWT estático emitido el 16/07/2020, sin expiración, hardcodeado en el frontend. Hashes bcrypt de 272 usuarios expuestos. Sin rotación de tokens. Sin MFA.

Remediación: OAuth2/OIDC con tokens de corta duración (15 min), refresh tokens, rotación. Nunca exponer hashes.

VIOLADO

API3:2023 — Broken Object Property Level Authorization

Evidencia: Las APIs devuelven TODOS los campos sin filtrar. /crg/usuario devuelve "Clave" (hash bcrypt). /crg/operador devuelve correos, celulares, direcciones privadas.

Remediación: DTOs/ViewModels con solo campos necesarios. Projections a nivel de base de datos.

PROBABLE

API4:2023 — Unrestricted Resource Consumption

Evidencia: Sin rate limiting. Respuestas completas sin paginación (194 operadores, 272 usuarios, 1122 municipios en un solo request).

Remediación: Rate limiting (X-RateLimit headers), paginación obligatoria, timeouts.

VIOLADO

API5:2023 — Broken Function Level Authorization

Evidencia: Un token con rol "intelligence" (BI/lectura) accede a endpoints administrativos como /crg/usuario que expone datos de administradores del sistema.

Remediación: RBAC granular. Separar APIs públicas, de operador y administrativas.

PROBABLE

API6:2023 — Unrestricted Access to Sensitive Business Flows

Evidencia: Es posible descargar la base de datos completa de estaciones, operadores y usuarios en pocas llamadas HTTP, replicando toda la información del sistema.

Remediación: Anti-scraping, CAPTCHA en endpoints masivos, monitoreo de patrones.

VIOLADO

API8:2023 — Security Misconfiguration

Evidencia: CORS permisivo, JWT sin expiración hardcodeado en frontend, servidor en puerto no estándar (3011), sin headers de seguridad (CSP, HSTS, X-Content-Type-Options).

Remediación: CORS restrictivo, HSTS, CSP, API Gateway, secretos en vault.

VIOLADO

API9:2023 — Improper Inventory Management

Evidencia: Endpoints no documentados descubiertos por enumeración (/crg/operador, /crg/usuario, /crg/consentimiento). Sin documentación ni inventario público de APIs.

Remediación: Documentar con OpenAPI/Swagger, implementar API Gateway con inventario.

✅ Recomendaciones de Remediación

Priorizadas por criticidad e impacto. Estas recomendaciones pueden incluirse en la comunicación formal al MinEnergía.

🔴 INMEDIATAS (0-30 días)

1.

Remover campo "Clave" de la respuesta de /crg/usuario

Nunca se deben exponer hashes de contraseña vía API. Implementar un DTO que excluya este campo. Esfuerzo estimado: 1 hora.

2.

Revocar el token JWT estático actual

Generar tokens dinámicos con expiración (15 min access + refresh token). Remover el token del código frontend.

3.

Restringir /crg/usuario y /crg/consentimiento solo a rol Administrador

Estos endpoints no deben ser accesibles con un token de lectura/BI. Implementar validación de rol en el backend.

4.

Filtrar campos sensibles de /crg/operador

La API pública solo debe devolver: Nombre, Siglas, TipoOperador, Estado, Departamento, Sitio. Remover: Correo, Celular, Dirección, NIT, Teléfono.

🟠 CORTO PLAZO (1-3 meses)

5.

Implementar OAuth2 / OpenID Connect

Reemplazar JWT estático por flujo OAuth2 con servidor de autorización. Tokens con expiración, refresh tokens, scopes por tipo de usuario.

6.

Implementar RBAC (Role-Based Access Control) granular

Definir roles: Público (solo estaciones), Operador (sus datos), Admin (todo). Cada endpoint valida permisos.

7.

Configurar CORS restrictivo

Solo permitir origin https://cargame.minenergia.gov.co y dominios autorizados de consumidores de API.

8.

Rate limiting y paginación

Límite de requests por IP/token. Paginación obligatoria en endpoints masivos. Headers X-RateLimit-*.

9.

Headers de seguridad HTTP

Implementar: Strict-Transport-Security, Content-Security-Policy, X-Content-Type-Options, X-Frame-Options.

🟡 MEDIANO PLAZO (3-6 meses)

10.

Implementar API Gateway

Centralizar autenticación, rate limiting, logging, y documentación. Inventario de endpoints activos.

11.

Documentación OpenAPI/Swagger

Documentar todas las APIs formalmente. Permitir acceso controlado a consumidores autorizados como UPME.

12.

Auditoría de seguridad (Penetration Testing)

Contratar pentesting profesional para evaluar endpoints POST/PUT/DELETE, inyección SQL, XSS, y otros vectores no cubiertos en este análisis.

13.

Actualización de Política de Tratamiento de Datos

Revisar la política de protección de datos personales de CARGAME conforme a la Ley 1581 y registrar ante la SIC.

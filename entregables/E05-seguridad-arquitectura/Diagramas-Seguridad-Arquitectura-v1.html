Visión General de Seguridad

Modelo de seguridad Zero Trust para la plataforma de interoperabilidad de electromovilidad. Flujo completo desde registro de CPO hasta consumo seguro de APIs OCPI 2.2.1.

Diagrama de Contexto (C4 Nivel 1) — Ecosistema de Seguridad

Flujo de Conexión en 4 Pasos — Seguridad y Habilitación

1
Registro (Cárgame)
Reporte inicial de infraestructura en la plataforma del Ministerio. CPO queda registrado con NIT y estado ACTIVE.
2
Notificación UPME
MME valida y notifica oficialmente a la UPME vía API webhook. Se crea registro preliminar del CPO.
3
Emisión API Key
UPME genera credenciales únicas con alcances asignados vía canal seguro. KeyCloak crea OAuth Client.
4
Generación Token JWT
El CPO usa el API Key para solicitar un token de vida corta (OAuth 2.1 / mTLS) para iniciar el reporte.
Principio: Autenticación mutua y cifrado en tránsito/reposo garantizan integridad, trazabilidad y no repudio. Toda la gestión de identidad se centraliza en KeyCloak HA (2 pods en OKE, realm dedicado por tipo de actor).

9 Capas de Defensa — Zero Trust

L1-L3: Perímetro
  • OCI WAF: OWASP Top 10, DDoS L7, GeoFence CO
  • OCI DNS: Traffic Steering, Health Checks
  • VCN + NSGs: Segmentación pub/priv, Flow Logs
  • Load Balancer: TLS 1.3 termination
L4-L6: Identidad
  • API Gateway: Rate Limit + Token Introspection
  • KeyCloak HA: OAuth 2.1, RBAC, Multi-Realm
  • OCI Certificates: mTLS X.509 por CPO
  • Service Mesh: mTLS auto inter-servicio
L7-L9: Datos & Audit
  • OCI Vault: CMK, rotación 90d, JWT signing keys
  • ATP TDE: Cifrado transparente datos
  • WORM Audit: Logs inmutables (SIC)
  • Cloud Guard: Postura continua, auto-remediation

Principios Arquitectónicos para la Integración CARGAME - UPME

Estos principios son obligatorios para garantizar que la integración entre CARGAME y la plataforma UPME cumpla con los estándares de seguridad y escalabilidad requeridos por la Resolución 40559/2025.

P1: Zero Trust — Nunca Confiar, Siempre Verificar
  • Toda comunicación CARGAME → UPME debe ser mutuamente autenticada
  • Tokens con TTL máximo de 15 minutos, sin refresh para M2M
  • Cada request validado independientemente (sin sesiones implícitas)
  • VPN Site2Site IPSec como canal exclusivo para verificación de CPOs
  • Eliminar JWT estático: migrar a OAuth 2.1 con rotación automática
P2: Gestión Centralizada de Identidades
  • KeyCloak HA como IdP único para toda la plataforma
  • Realm dedicado upme-cpo para gestión de 500+ operadores
  • Client Credentials + Certificate Binding (RFC 8705) para M2M
  • Token Introspection (RFC 7662) para validación en API Gateway
  • Revocación inmediata si CARGAME reporta CPO suspendido
P3: Cifrado End-to-End y Protección de Datos
  • TLS 1.3 obligatorio en todo tránsito (eliminar protocolos legacy)
  • mTLS entre microservicios via OCI Service Mesh
  • Datos en reposo: TDE + CMK en Autonomous DB, AES-256 en Vault
  • Audit logs inmutables en WORM Object Storage
  • PII clasificada y protegida según Ley 1581/2012
P4: Escalabilidad y Resiliencia
  • 500 CPOs × 60s = 21.6M transacciones/mes proyectadas
  • Circuit Breaker para aislar fallos de CARGAME (Open 5 fallos, half-open 30s)
  • Cache Redis 24h para reducir dependencia de CARGAME
  • OCI Streaming (Kafka) para procesamiento asíncrono y desacoplado
  • OKE Autoscaling 1→3 nodos para absorber picos de carga
P5: Integración Segura por Eventos
  • Webhooks firmados con HMAC-SHA256 (shared secret en OCI Vault)
  • Verificación de idempotencia por delivery_id (24h TTL en Redis)
  • IP Whitelist en WAF para origenes autorizados de CARGAME
  • Doble verificación: Webhook + consulta VPN para cada CPO nuevo
  • Dead Letter Queue para eventos fallidos con retry exponencial
P6: Observabilidad y Auditoría Continua
  • Correlation ID de extremo a extremo en cada transacción
  • Audit trail inmutable: login, token, API call, lifecycle CPO
  • Grafana + Prometheus + Loki en OKE para monitoreo RT
  • Alertas automáticas: intentos fallidos, anomalías, SLA breach
  • Reportes de cumplimiento para la SIC y el Ministerio

Recomendaciones para el Ministerio de Minas y Energía

Para garantizar la interoperabilidad segura y escalable entre CARGAME y la plataforma nacional UPME, se recomienda al Ministerio:

Corto Plazo (0-3 meses)
  • Eliminar JWT estático y migrar a autenticación dinámica
  • Proteger endpoint /crg/usuario que expone hashes de contraseñas
  • Implementar CORS restrictivo con origins autorizados
  • Establecer VPN Site2Site con UPME para canal seguro
  • Definir contrato de APIs con versionamiento y OpenAPI spec
Mediano Plazo (3-6 meses)
  • Implementar OAuth 2.1 con tokens de vida corta para APIs
  • Agregar WAF con reglas OWASP Top 10
  • Rate limiting por consumidor con cuotas definidas
  • Cifrar PII en reposo y enmascarar en respuestas API
  • Webhook bidireccional con firma HMAC para eventos CPO

Capas de seguridad en arquitectura UPME
Conclusión: La plataforma UPME está diseñada con arquitectura de seguridad enterprise-grade (9 capas, Zero Trust, KeyCloak HA, mTLS, OCI Vault). Sin embargo, la integración con CARGAME introduce riesgos que deben mitigarse conjuntamente. UPME implementa protecciones defensivas (VPN, Circuit Breaker, cache, validación), pero es fundamental que CARGAME evolucione su postura de seguridad para garantizar la integridad de los datos que alimentan la política nacional de electromovilidad de Colombia.

Para el equipo Oracle Architecture:

2. Contexto del Proyecto

2.1 Regulación & Mandato

Emisor: Ministerio de Minas y Energía — Fecha: 21 Nov 2025
Resolución: 40559/2025 — Lineamientos de interoperabilidad para estaciones de carga de VE de acceso público
Estándar adoptado: OCPI 2.2.1 (Open Charge Point Interface)
Módulos obligatorios: Locations, EVSEs, Tariffs, Sessions, CDRs, Tokens, Commands
Entidad responsable: UPME (entidad pública del orden nacional)

La UPME debe construir y operar la plataforma que habilite la interoperabilidad de carga de VE en Colombia. Cualquier usuario pueda cargar en cualquier estación pública, independientemente del operador, con transparencia en precios y disponibilidad en tiempo real.

3. Arquitectura Actual (Línea Base)

Principio rector: Simplicidad operacional sobre escalabilidad prematura. Arquitectura dimensionada para el perfil real (~10 usuarios, ~10-50 CPOs) con prioridad en cumplir plazos regulatorios con equipo de 8 personas.

3.1 Principios de Diseño

Principios Arquitectónicos

  • Right-sizing: Dimensionado al volumen real
  • Monolito sincrónico: Un servicio FastAPI
  • CRUD simple: Sin CQRS ni Event Sourcing
  • Managed services first: Maximizar servicios OCI gestionados
  • Security by design: Capas de seguridad desde día 1
  • API-first: Contract-first con OpenAPI 3.0

Decisiones Clave (ADRs)

  • ADR-001: Python FastAPI (monolito) vs microservicios
  • ADR-002: Container Instances vs OKE
  • ADR-003: Procesamiento síncrono vs Kafka/Streaming
  • ADR-004: Object Storage prefijos vs Data Lake Medallion
  • ADR-005: OCI Monitoring nativo vs Prometheus/Grafana
  • ADR-006: 4 ambientes vs 6

Recomendaciones de Seguridad

SEC-01 Habilitar OCI Cloud Guard — Monitoreo continuo de postura de seguridad. Detecta configuraciones inseguras y responde automáticamente. Costo: Incluido en OCI.
SEC-02 OCI Vulnerability Scanning Service — Escaneo de vulnerabilidades en Container Instances y hosts. Complementa OCIR scan. Costo: Incluido.
SEC-03 Customer-Managed Keys (CMK) via OCI Vault — Para datos críticos de auditoría SIC, control total del ciclo de vida criptográfico. Costo: ~$0.53/key version/mes.
SEC-04 OCI Security Zones — Prevenir configuraciones inseguras en compartimento de producción (ej: bucket público, DB sin cifrado). Costo: Incluido.
SEC-05 OCI IAM Identity Domains — Complementar KeyCloak con federación al directorio corporativo UPME. Free tier disponible.

Recomendaciones de Confiabilidad

REL-01 Múltiples Container Instances + LB — Desplegar 2+ instancias en AD distintos. HA activo-activo para la capa de app. Elimina single-point-of-failure de compute.
REL-02 OCI Health Checks externos — Monitoreo end-to-end desde múltiples regiones + OCI Notifications para alertas. Costo: Incluido.
REL-03 Runbooks de recuperación — Procedimientos documentados por componente. DR drills trimestrales. Automatizar con OCI Resource Manager stacks.
REL-04 Path a OKE — Cuando CPOs >50 o pods >4, migrar a OKE (1-2 nodos) para auto-scaling y self-healing. Path Container Instances → OKE es claro en OCI.

Recomendaciones de Performance & Costos

PERF-01 Object Storage Lifecycle Policies — Datos >90d → Infrequent Access, >1a → Archive. Ahorro ~60-80% en datos históricos de auditoría.
PERF-02 OCI Budgets & Cost Analysis — Alertas al 80% y 100% presupuesto mensual. Tracking por compartimento. Incluido.
PERF-03 Métricas custom aplicación — Instrumentar FastAPI: latencia por endpoint OCPI, errores validación, tiempo Cárgame. OCI Monitoring Ingestion API.
PERF-04 ATP Performance Hub — Análisis queries lentos, wait events, optimization advisor automático. Incluido en ATP.

Recomendaciones Operacionales

OPS-01 OCI Logging Analytics — Búsqueda full-text, correlación de logs, dashboards, retención configurable. Esencial para troubleshooting CPO. ~$0.60/GB (10 GB gratis/mes).
OPS-02 OCI Application Performance Monitoring (APM) — Tracing distribuido, mapeo dependencias, cuellos de botella. Valioso para VPN → Cárgame. ~$0.10/100 spans.
OPS-03 Dashboards SLO — Disponibilidad API (99.9%), latencia P99, tasa validación OCPI, freshness datos. OCI Monitoring + Notifications.
OPS-04 Proceso formal incidentes — P1 (plataforma caída): 15 min. P2 (degradación): 1h. P3: 4h. Postmortems obligatorios P1/P2.

9.3 Convenciones de Diagramación

Reglas de estilo para diagramas Oracle:
  • Fondo: Blanco (#FFFFFF) con bordes de región/VCN en gris oscuro
  • Regiones OCI: Rectángulo con borde punteado, etiqueta "OCI Region: Bogotá"
  • VCN: Rectángulo sólido, subnets como sub-rectángulos internos
  • Servicios: Usar iconos oficiales del Toolkit v24.2
  • Conexiones: Líneas sólidas para datos, punteadas para monitoreo/control
  • Colores: Oracle Red (#C74634) para servicios de seguridad, azul para compute, verde para data
  • Etiquetas: Nombre del servicio + descripción breve (ej: "OCI WAF / OWASP Top 10")
Entregable pendiente: Los diagramas formales en formato draw.io (usando el OCI Architecture Diagram Toolkit v24.2) se entregarán como adjunto separado. Los diagramas Mermaid incluidos en este documento representan la vista lógica que será traducida a formato Oracle oficial.

11. Solicitud al Equipo Oracle Architecture

Áreas de Colaboración Solicitadas

La UPME busca establecer una partnership estratégica con Oracle para el fortalecimiento de la arquitectura de la plataforma de interoperabilidad de electromovilidad. Solicitamos al equipo de Oracle Architecture su apoyo en las siguientes áreas:

11.1 Validación de Arquitectura

  • Revisión y validación del assessment WAF presentado en este documento
  • Feedback sobre las decisiones arquitectónicas (ADRs) y el approach de right-sizing
  • Validación de la estrategia de seguridad (mTLS + OAuth 2.0 + WAF + Vault)
  • Review del diseño de VCN y segmentación de red

11.2 Acompañamiento en Implementación

  • Asesoría en la configuración del Landing Zone según CIS Benchmark
  • Best practices para OCI Container Instances en producción (HA, scaling, networking)
  • Configuración óptima de Autonomous Database para carga OCPI
  • Implementación de OCI Cloud Guard, Security Zones y Vulnerability Scanning
  • Guía de migración futura: Container Instances → OKE

11.3 Optimización de Costos

  • Review del sizing actual y recomendaciones de instancias
  • Evaluación de Committed Use Discounts aplicables
  • Programas de créditos o soporte para entidades públicas (si disponibles)
  • Optimización de costos de Object Storage con lifecycle policies

11.4 Soporte Continuo

  • Acceso a Oracle Architecture Center para consultas de referencia
  • Sesiones de architecture review periódicas (sugerimos: pre-Sandbox y pre-Go-Live)
  • Canal de comunicación directo con el equipo de Oracle Architecture LATAM
  • Participación en el programa Oracle Customer Success para seguimiento post Go-Live
